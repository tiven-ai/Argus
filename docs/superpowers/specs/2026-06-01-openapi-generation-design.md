# OpenAPI 生成设计（从 Zod schema 生成 + 防漂移）

> 日期：2026-06-01
> 状态：已批准方案，待写实现计划
> 范围：`apps/server` 新增生成脚本 + 注册表 + 测试；`docs/api/openapi.yaml`（生成物入库）；CI 漂移检查。不重构现有运行时校验。
> 关联：兑现 [`docs/api/README.md`](../../api/README.md) 已写下的"OpenAPI 从 Zod 生成、CI 发布"承诺；防腐方式沿用 `docs/superpowers/specs/2026-06-01-docs-anti-rot-ci-design.md` 的"重新生成 + git diff 失败"模式。

## 背景与目标

`docs/api/README.md` 声称"The full OpenAPI spec is generated from Zod schemas in apps/server and published to openapi.yaml by CI"——但 `openapi.yaml`、生成器、CI 步骤目前都**不存在**，是 aspirational 文案。本项把它变成真实：从已有的 Zod schema 生成一份 OpenAPI 3 spec，提交入库，并用 CI 守住不漂移。

## 数据现状（前提，经代码勘察）

- 服务端 15 个 HTTP 路由（auth ×8、api ×2、projects ×4、tokens ×3、ingest ×1、healthz、SSE ×1），全部用**手动 `safeParse()`** 校验——Fastify 没有 schema 注册表可收割。
- Zod `^3.23.0`（`apps/server` 与 `packages/shared-types` 一致）。
- shared-types 已导出响应 schema：`ListSessionsResponseSchema`、`GetSessionResponseSchema`、`SessionSummarySchema`、`StepSchema` 等（`packages/shared-types/src/api.ts`）。
- ingest 的 OTLP 请求体 schema：`otlpExportRequestSchema`（`apps/server/src/modules/ingest/otlp-json.ts`，经 `modules/ingest/index.ts` 再导出）。
- 无任何现成 OpenAPI/swagger 工具依赖。
- 选用 `@asteasolutions/zod-to-openapi`（Zod 3 兼容，registry 式，单一真源）。

## 范围（仅公开 API）

只纳入客户端真正面向的端点：

| 端点                           | 请求 schema                         | 响应 schema                                          |
| ------------------------------ | ----------------------------------- | ---------------------------------------------------- |
| `POST /v1/traces`              | `otlpExportRequestSchema`（ingest） | `{ accepted: number }`（registry 内联）              |
| `GET /api/sessions`            | query：`limit?`、`projectId?`       | `ListSessionsResponseSchema`（shared-types，已存在） |
| `GET /api/sessions/:sessionId` | path：`sessionId`                   | `GetSessionResponseSchema`（shared-types，已存在）   |
| `GET /healthz`                 | —                                   | `{ status: 'ok' }`（registry 内联）                  |

**不纳入**：auth/projects/tokens CRUD（web app 内部用，非外部客户端面向）、SSE `/api/sessions/:id/stream`（`text/event-stream`，OpenAPI 表达不佳，README 已标注另见 `sse-protocol.md`）。

## 架构（独立生成脚本 + zod-to-openapi）

- `apps/server` 加依赖 `@asteasolutions/zod-to-openapi`。
- 新建 `apps/server/src/openapi/registry.ts`：
  - 创建 `OpenAPIRegistry`，`registerPath(...)` 注册上面 4 个端点。
  - 复用 shared-types 已有的 schema（OTLP body、sessions 响应）。`zod-to-openapi` 需要 schema 带 `.openapi()` 元数据或经 `extendZodWithOpenApimetadata` 处理——对从 shared-types 导入的现成 schema，用 registry 的 `register(name, schema)` 注册组件即可，无需改动 shared-types 源（不污染 shared-types，不在它那边引入 openapi 依赖）。
  - 仅为 ingest 响应 `{ accepted: number }` 与 healthz `{ status: 'ok' }` 这两个单行响应在 registry 文件里就近定义极简 Zod schema（不值得进 shared-types）。
  - 导出一个纯函数 `buildOpenApiDocument(): OpenAPIObject`（registry → `OpenApiGeneratorV3().generateDocument(...)`，含 info.title/version、servers）。
- 新建 `apps/server/src/cli/gen-openapi.ts`：调 `buildOpenApiDocument()` → YAML 序列化 → 写 `docs/api/openapi.yaml`。
- `apps/server/package.json` 加脚本 `"gen:openapi": "tsx src/cli/gen-openapi.ts"`，根可用 `pnpm --filter @argus/server gen:openapi`。
- `docs/api/openapi.yaml` 提交入库（人和工具可直接消费）。

**单一真源**：请求/响应结构来自 Zod schema（多数已在 shared-types）；`openapi.yaml` 是派生物，不手改。

## 防腐（重新生成 + git diff 失败）

CI（`.github/workflows/ci.yml`）加一个与现有 job 并行的 `openapi-drift`：

- checkout → pnpm/action-setup → setup-node(20, cache pnpm) → `pnpm install --frozen-lockfile`
- `pnpm --filter @argus/server gen:openapi`
- `git diff --exit-code docs/api/openapi.yaml` —— 若 schema 改了但没重新生成，diff 非空 → job 红。

这正是"generated by CI"的真实保证。现有 `ci`、`docs-links`、`docs-build` job 不动。

## 测试

- `apps/server/test/openapi/registry.test.ts`（新增，vitest，纯函数、不落盘）：
  - 调 `buildOpenApiDocument()`，断言：`openapi` 字段为 `3.x`；`paths` 含 `/v1/traces`（post）与 `/api/sessions`、`/api/sessions/{sessionId}`（get）与 `/healthz`；`/v1/traces` 的 requestBody 引用了 OTLP 组件；`/api/sessions` 的 200 响应引用了 sessions 列表组件。
  - 断言**不含** auth/projects/tokens/stream 路径（范围守卫）。

## 验收

- `pnpm --filter @argus/server gen:openapi` 产出 `docs/api/openapi.yaml`，是合法 OpenAPI 3。
- 重新跑生成器，`git diff --exit-code docs/api/openapi.yaml` 为空（幂等）。
- 新测试通过；既有 `pnpm test` / `typecheck` / `lint` / `build` 不受影响。
- CI 四个 job（ci / docs-links / docs-build / openapi-drift）全绿。
- 故意改一个纳入的 schema（如给 OTLP 加字段）但不重新生成 → `openapi-drift` 红；重新生成后绿。

## 非目标（留作后续）

- 不纳入 auth/projects/tokens/SSE 端点。
- 不重构现有 `safeParse` 运行时校验（生成与校验解耦）。
- 不把 `openapi.yaml` 渲染进 VitePress 门户（门户当前 srcExclude 了 `api/`；接入是干净的后续项）。
- 不引入 `@fastify/swagger` 运行时 spec。
